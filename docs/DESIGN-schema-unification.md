# Уніфікація схеми `.a7p` через JSON Schema + Quicktype

## Проблема

`profile.clj` — оригінальна Clojure-специфікація (ArcherBC2), що описує обмеження полів
профілю в "людських" одиницях (double + `fraction-digits`, напр. `r-twist` 0.00–100.00 in/turn).

Три репозиторії (`py`, `js`, `dart`) незалежно реалізують той самий набір правил
у трьох різних валідаційних DSL, перерахувавши межі в raw-цілі (wire-формат, ×10^n):

| Репо | Файл | DSL |
|---|---|---|
| py | `src/a7p/yupy_schema.py` | yupy |
| js | `src/validate.ts` | yup |
| dart | `lib/src/a7p_field_constraints.dart` + `a7p_validator.dart` | ручні Dart-класи |

Дублювання вже призвело до дрейфу: `A7pFieldConstraints.zoom` у Dart свідомо відхиляється від
`profile.clj` (0–4 → 1–6, задокументовано як "баг оригіналу"), тоді як інші репо цього фікса
можуть не мати. Кожна майбутня зміна вимагає синхронного ручного патчу в 3 місцях.

Proto-файл (`profedit.proto`) вже є спільним джерелом для *форми* повідомлення (він буквально
скопійований у всі три репо), але не несе інформації про діапазони/довжини/умовні правила —
цим займається кожна мова окремо.

## Ціль

Одне джерело правди — JSON Schema — що описує одночасно форму даних і всі обмеження, з якого:

1. **Типи** генеруються Quicktype-ом для кожної мови.
2. **Валідація** виконується нативним JSON-Schema-валідатором відповідної мови напряму проти
   цієї ж схеми — замість ручного портування правил у yup/yupy/Dart-класи.

### Що реально дає Quicktype (і чого не дає)

Quicktype читає JSON Schema і генерує interfaces/classes + (де підтримується) serialization
(`fromJson`/`toJson`). Він **не** транслює `minimum`/`maximum`/`maxLength`/умовні `if/then` у
код, що виконує перевірки під час рантайму — ці ключові слова в кращому разі осідають у
докстрінгах. Тобто Quicktype закриває лише частину "уніфікації типів", а не "уніфікацію
валідації". Валідацію потрібно виконувати окремим, стандартним JSON-Schema-валідатором,
який читає ту саму схему безпосередньо (без кодогену):

| Мова | Бібліотека-валідатор | Замінює |
|---|---|---|
| TypeScript / JS | `ajv` | `validate.ts` (yup) |
| Python | `fastjsonschema` (не `jsonschema`, див. бенчмарк нижче) | `yupy_schema.py` |
| Dart | `json_schema` (pub.dev) | `a7p_field_constraints.dart` + `a7p_validator.dart` |

### Бенчмарк на реальних даних (484 файли з `a7p-lib/gallery` + тестові фікстури)

Перевірено на практиці (`schema/a7p.schema.json` + ручна перевірка унікальності `mv`, що
документована як не-виражувана в чистому JSON Schema, — без неї недорахунок 52 файлів):

| Валідатор | us/файл | Відносно `yupy` | Результат (invalid/484) |
|---|---|---|---|
| `yupy` (поточний, ручний) | 623 | 1x (базова лінія) | 52 |
| `jsonschema` (pure-Python) | 2702 | **0.23x — у 4.3 рази повільніше** | 52 (з ручним mv-check) |
| `fastjsonschema` (компільований) | 116 | **5.4x швидше** | 52 (з ручним mv-check) |

Усі три дають ідентичний набір з 52 невалідних файлів (усі — з `.unvalidated/`, усі з причини
дубльованих `mv`) — тобто схема з одним невеликим допоміжним чеком повністю відтворює поведінку
поточного `yupy`-валідатора, а `fastjsonschema` ще й у 5+ разів швидший за нього. Голий
`jsonschema`, попри простоту, для Python — гірший вибір, ніж здавалося на етапі дизайну.

Це ключова відмінність від "перенести yup/yupy-схему 1:1" — замість трьох ручних портів
валідаційної логіки лишається один JSON-файл і три тонкі виклики `validate(schema, data)`.

## Структура схеми

```
a7p-cross/
  schema/
    a7p.schema.json       # єдине джерело правди (draft 2020-12)
    generate.sh            # quicktype-виклики для ts/py/dart
    fixtures/
      valid/*.json         # набір валідних payload'ів
      invalid/*.json        # набір невалідних + очікувана помилка/поле
```

`a7p.schema.json` описує `$defs`: `Payload`, `Profile`, `SwitchProps`, `CoefRow`, плюс enum-и
`TwistDir`, `BcType`, `DistanceFrom`. Числові поля використовують `minimum`/`maximum` у
raw-цілих (той самий "wire" масштаб, що й зараз у yupy/yup/Dart — це вже перевірений канонічний
рівень, а не одиниці з `profile.clj`).

### Vendor-розширення (`x-*`)

Стандартний JSON Schema не покриває все, що зараз явно закодовано в Dart-версії
(`A7pFieldRange`) для UI (одиниці виміру, кількість знаків після коми для відображення) чи
кастомну семантику ("unused"-сентинели, унікальність `mv` окрім нуля). Ці метадані виносяться
в `x-`-префіксовані ключі, які валідатори ігнорують, але codegen/UI-шар може читати:

```jsonc
"rTwist": {
  "type": "integer", "minimum": 0, "maximum": 10000,
  "x-fraction-digits": 2, "x-unit": "in/turn"
},
"coefMv": {
  "type": "integer",
  "x-fraction-digits": 1, "x-ui-fraction-digits": 0, "x-unit": "m/s"
}
```

Це прямий еквівалент сьогоднішнього `A7pFieldConstraints` у Dart — але генерований, а не
ручний, і той самий файл читають усі три мови.

### Умовні правила (coefRows залежно від bcType)

JSON Schema `if`/`then` покриває розгалуження G1/G7 (max 5 рядків, `mv` ≤ 30000) проти CUSTOM
(max 200 рядків, `mv`("ma") ≤ 100000):

```jsonc
"if": { "properties": { "bcType": { "const": "CUSTOM" } } },
"then": { "properties": { "coefRows": { "maxItems": 200, "items": { "properties": { "mv": { "maximum": 100000 } } } } } },
"else": { "properties": { "coefRows": { "maxItems": 5,   "items": { "properties": { "mv": { "maximum": 30000  } } } } } }
```

### Що JSON Schema НЕ виражає нативно

- **Унікальність `mv` окрім значення 0** — не покривається стандартним `uniqueItems` (він
  вимагає унікальності всіх елементів). Лишається невеликою, один раз написаною, перевіркою
  на кожній мові (як і зараз) — але оформленою як єдиний документований інваріант
  (`x-unique-except-zero: "mv"` як анотація-документація), а не приховане правило, яке треба
  здогадатися продублювати.
- **"unused"-сентинели** (`cIdx == 255` означає "не використовується", `distance == 0` означає
  "unused") — виражаються через `oneOf`, але семантика ("чому 255/0 особливі") варта
  коментаря/анотації, а не лише схеми.

## Пайплайн генерації

1. `schema/a7p.schema.json` редагується вручну (це і є новий "profile.clj").
2. `generate.sh` викликає `quicktype schema/a7p.schema.json -o ...` для кожного репо/мови,
   кладучи результат у місце, де зараз лежать ручні типи (`js/src/types.ts`,
   аналогічний dataclass/pydantic-файл у `py`, `.dart`-модель у `dart` — якщо
   вирішено не покладатись on protobuf-згенерованих класів для цього шару).
3. Кожен репо у себе просто читає `schema/a7p.schema.json` (або його копію/пакет) і викликає
   `ajv` / `jsonschema` / `json_schema` замість власного yup/yupy/Dart-валідатора.
4. `schema/fixtures/` — спільний набір валідних/невалідних payload'ів, які кожен репо
   під'єднує до свого тестового рантайму (pytest/jest/dart test), гарантуючи, що всі три
   реалізації погоджуються на одних і тих самих edge-cases (у т.ч. регресійний тест на
   випадок `zoom` 0–4 vs 1–6, який зараз розходиться).

**Статус (реалізовано):** `scripts/compile.py --python` — окремий, вужчий за Quicktype
крок: не генерує типи, а компілює сам валідатор (`fastjsonschema.compile_to_code()`) у
`py/src/a7p/_compiled_schema.py`, щоб прибрати ~20ms компіляції схеми на кожен
`import a7p`. Деталі, цифри й порядок регенерації — `schema/README.md`. `--ts`/`--dart` поки
що заглушки (потребують спершу міграції js на ajv і вибору аналога для dart).

## Рішення щодо відкритих питань

1. **Канонічні межі там, де репо вже розійшлися.** `zoom`: канонічний діапазон — **0–6**
   (0 = не налаштовано/дефолт для switch з `distance_from=INDEX`, підтверджено реальними
   профілями `example2.a7p` в обох `py`/`js`; 1–6 — реальні рівні зуму пристрою), а не
   `profile.clj`-івський 0–4 (документований баг оригіналу). Зафіксовано в
   `schema/a7p.schema.json` (`SwitchProps.zoom`).

   Усі виявлені баги-розбіжності (`c_muzzle_velocity`, `b_weight`, `b_length`, `c_zero_w_pitch`
   відсутній у Dart-валідаторі, `bc_cd`/`mv` у js занижені в 10 разів, `zoom` необмежений у
   js) — **виправлено** в `js`/`dart` (див. `x-fixed` в схемі для деталей по
   кожному полю).

   Два питання, що були не просто числовими розбіжностями, а різницею в семантиці
   валідації, тепер мають прийняте рішення (канон = поведінка `profile.clj`/`py`,
   сам `profile.clj` не чіпали):
   - **`c_idx`**: 0–200 або сентинел 255 ("unused"); 201–254 — невизначений проміжок і має
     відхилятись. `js`/`dart` досі приймають весь діапазон 0–255 без винятку —
     **не виправлено** (це зміна логіки валідації, а не константи).
   - **`switches[].distance`**: 0 ("unused") або `[100, 300000]`, **незалежно** від
     `distance_from` — на відміну від `js`/`dart`, які трактують `distance` як індекс
     0–255 при `distance_from=INDEX`. **Не виправлено** там же з тієї ж причини.

   Обидва рішення закодовані як `x-decision` в `schema/a7p.schema.json` і замкнені тестами
   `py/tests/test_schema_validator.py` (`test_switch_c_idx_bounds`,
   `test_switch_distance_bounds_independent_of_distance_from`) — які на відміну від
   старішого `test_yupy_schema.py` перевіряють саме публічний шлях `a7p.validate()`
   (`schema_validator.py`), а не вимкнений з публічного API `yupy_schema.py`.
2. **та 4. Розповсюдження схеми/fixtures.** `a7p-cross` стає монорепо — `py`, `js`,
   `dart` переїжджають усередину як пакети одного репозиторію. Це знімає питання
   копіювання/submodule/npm-залежності: `schema/a7p.schema.json` і `schema/fixtures/`
   лежать в одному дереві поруч з усіма трьома пакетами, кожен referencing їх відносним
   шляхом напряму (як зараз `profedit.proto` дублюється по копії — у монорепо копії більше
   не потрібні, це має бути один файл).
3. **Публічне API кожного пакета бажано не міняти.** Це означає: Quicktype-згенеровані типи
   **не замінюють** існуючі публічні класи/інтерфейси (`types.ts` у js, публічні
   dataclass/типи в py, публічні класи в dart) — щонайменше не в рамках цієї
   уніфікації. Практичний наслідок для архітектури:
   - Схема одразу замінює **валідаційний шар** (yupy/yup/Dart-класи → ajv/jsonschema/json_schema
     проти `schema/a7p.schema.json`) — це внутрішня деталь реалізації, публічного API не
     торкається (сигнатура `validate(payload)` лишається).
   - Quicktype-генерація типів стає **опційним** кроком і, якщо застосовується, генеровані
     типи мають бути 1:1 сумісні з існуючими публічними (ті самі назви полів/enum-значень,
     той самий casing) — або взагалі не виставляються назовні, а лишаються внутрішнім шаром,
     а публічні класи й далі пишуться вручну (чи генеруються з protobuf, як зараз) і просто
     мапляться в/з internal-схема-типу.
   - Тобто зі старту практичний обсяг — це заміна валідації, а не типів; ризикована частина
     (заміна публічних типів) відкладається до окремого рішення, якщо взагалі знадобиться.
